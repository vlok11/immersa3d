---
description: Immersa 3D 代码生成约束与工程规范 (Professional Standard)
---

为确保 Immersa 3D 项目长期的可维护性、极高的渲染性能以及架构的纯净度，所有代码生成必须严格遵循以下约束。严禁产出过程式逻辑（Spaghetti Code）与非模块化代码。

1. 架构约束：严苛的“主从模块”模式
   强制解耦: 每个 PrimaryModule 必须拥有独立的文件夹。禁止主模块间直接引用对方的内部私有状态。
   状态单向流: 模块内部状态通过局部 Zustand 处理；全局状态通过 AppStore 转发。
   生命周期守卫: 任何 3D 资源的创建（Geometry, Texture, Shader）必须在 ImmersaModule.setup 中初始化，并在 dispose 中明确释放显存。
2. React & R3F (React Three Fiber) 开发准则
   声明式优先: 严禁在组件生命周期外部手动操作 DOM 或控制器，除非是在 useFrame 的高频回调中。
   Render Loop 隔离:
   严禁在页面级组件中使用 useFrame。
   useFrame 必须仅存在于原子级 3D 组件（如 DepthMesh.tsx）中。
   性能红线:
   所有 3D 对象必须使用 useMemo 或 useLoader 预加载。
   严禁在渲染循环中实例化 new THREE.Vector3() 或 new THREE.Color()。必须使用外部常量或对象池。
3. TypeScript 严格类型约束
   禁止 Any: 全局开启 no-explicit-any。所有 API 返回值、Shader 传递的 Uniforms 必须定义对应 Interface。
   模块上下文定义: 所有子模块必须显式声明其所属父模块的类型，确保 parent 方法调用的提示性。
   枚举规范: 禁止使用数字枚举，必须使用字符串枚举（String Enums）以增强日志可读性。
4. WebGPU & GLSL Shader 规范
   Shader 模块化: 复杂的 GLSL 必须拆分为 .glsl 或 .frag/.vert 文件。禁止在 JSX 中写超过 20 行的字符串 Shader。
   Uniforms 命名: 必须遵循 uTime, uResolution, uMap 等加前缀的命名规范。
   变量注释: 每个 Shader 计算的核心公式（如 Sobel 算子、位移算法）必须配有简洁的英文物理意义注释。
5. 命名美学与工程目录
   文件命名:
   组件: PascalCase.tsx
   逻辑/常量: camelCase.ts
   样式: module.css (强制开启 CSS Modules)
   目录结构: 严禁在 src/ 根目录直接放置业务文件。必须按 core/, modules/, shared/, store/ 分层。
6. 状态管理 (Zustand) 最佳实践
   Store 切片制: 禁止创建一个万能的 useStore。必须按功能切片（如 useAnalysisStore, useViewportStore）。
   非 React 环境接入: 必须提供 _.getState() 与 _.setState() 接口，以便在 WebWorker 或原生 JS 类中操作状态。
7. AI 推理管线约束
   Worker 隔离: 所有模型推理 (session.run) 必须运行在独立的 Web Worker 中。
   显存预警: 初始化模型前必须进行底层环境检测（WebGPU 是否可用），并预留 10% 的显存缓冲阈值。
   WARNING

红线警示:

发现 dispose 逻辑缺失的代码将被视为不合格。
发现组件层级超过 4 层的“Props Drilling”（透传数据）将被重构。
发现同一文件内既有复杂 UI 逻辑又有重度 3D 渲染逻辑的，强制要求拆分。
Antigravity 将作为这份规范的终极执行者，确保输出的每一行代码都符合架构师的审美与工程标准。
